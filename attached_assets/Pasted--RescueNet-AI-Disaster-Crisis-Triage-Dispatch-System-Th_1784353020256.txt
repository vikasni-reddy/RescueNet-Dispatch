# RescueNet AI — Disaster Crisis Triage & Dispatch System
**Theme 3: Crisis Management, HealthTech & Emergency Response**

---

## 1. Why This Problem

During floods, cyclones, or building collapses (extremely common across India), affected people send help requests through many uncoordinated channels — calls, WhatsApp, SMS, walk-ins to relief camps. Responders (NGOs, disaster management cells, volunteers) get hundreds of messy, unstructured messages in the first "golden hours" and have to manually figure out: *what does this person need, how urgent is it, and who's closest to help?* That manual triage is slow, and delays cost lives.

This maps directly to the theme's "AI that works when people need it most," and it beats a generic chatbot idea because it combines **NLP extraction + urgency scoring + spatial matching** — three visibly different AI/logic layers a judge can actually see working, not just one wrapped LLM call.

**Why it scores well against the rubric:**
- **Problem & Impact** — narrow, concrete: "triage and dispatch during the first hours of a disaster," not "AI for disaster management."
- **AI Implementation** — the LLM does real structured extraction + reasoning, not decoration. Judges can open your code and see the prompt, the JSON schema, and the scoring logic.
- **Deployed Demo** — a live map + dashboard is inherently demo-friendly and visually convincing in a 2–3 min video.
- **Innovation** — most people will build "chatbot for X." A dispatch/matching engine is a level up.

## 2. The Problem Statement (use this in your Submission Form, Section 01)

> **Problem:** During disasters, incoming help requests arrive as unstructured free text from multiple channels, in mixed languages, with no consistent severity signal. Responders manually read and triage each one, which is slow and inconsistent, and resource allocation (which volunteer/ambulance goes where) is done ad hoc with no visibility into who's nearest or best suited.
>
> **Who it affects:** Disaster response NGOs, local disaster management authorities, and ultimately the affected population who wait longer for help because triage is manual.
>
> **Why existing solutions fail:** Most relief coordination still happens over spreadsheets, phone calls, and WhatsApp groups. There's no automatic urgency classification and no systematic nearest-resource matching — it depends entirely on whoever is reading messages at that moment.
>
> **Our approach:** RescueNet AI ingests free-text help requests (any language, any format), uses an LLM to extract structured fields (need type, urgency, people affected, key details), computes a priority score, plots every request on a live map, and recommends the best-matched available resource (volunteer team, ambulance, relief truck) by type-fit and distance — so a small response team can act on the most critical cases first instead of reading every message linearly.

## 3. Core Features

### MVP (build this first — non-negotiable for submission)
1. **Intake form** — free text box + optional location (map pin or address) + submit. This simulates the "SMS/WhatsApp/call center" input channel.
2. **AI extraction** — on submit, call an LLM to return structured JSON: `need_type`, `urgency`, `people_affected`, `summary`, `language_detected`.
3. **Priority scoring** — combine LLM urgency + keyword signals (e.g. "trapped," "unconscious," "children," "no food/water") + time-since-reported into a single priority score (0–100).
4. **Live dashboard**
   - Map (color-coded pins: red = critical, orange = high, yellow = medium, green = low)
   - Sorted priority queue/list view
   - Click a request → see extracted details + suggested nearest resource
5. **Resource pool** — a seeded list of volunteer teams/ambulances/relief trucks, each with a type (medical/rescue/food/shelter), capacity, and location.
6. **Matching logic** — for a selected request, rank available resources by (a) type match, (b) distance (haversine formula), (c) availability. One-click "Assign."
7. **Status pipeline** — Reported → Assigned → En Route → Resolved.

### Stretch (only if MVP is done and stable with time to spare)
- Real-time updates via Supabase Realtime instead of manual refresh.
- A small analytics panel: counts by type/urgency, average response time.
- Multilingual input demoed live (type in Hindi/Telugu, show it gets normalized).
- "AI reasoning" shown transparently in the UI (e.g., a one-line explanation of why a resource was recommended) — this is a great judge-facing touch and costs almost nothing to add.

**Do not build:** user auth/login, admin roles, payment, native mobile app. None of that helps you score higher and all of it burns time you don't have.

## 5. Where AI Actually Does the Work (be explicit about this in your Solution Description)

**Step 1 — Structured extraction (LLM call #1).** On every new request, send the raw text to the LLM with a system prompt like:

```
You are a disaster-response triage assistant. Given a free-text help
request (which may be in any language), extract and return ONLY valid
JSON with this schema:
{
  "need_type": "medical | rescue | food | water | shelter | other",
  "urgency": "critical | high | medium | low",
  "people_affected": <integer or null>,
  "summary": "<one sentence, in English>",
  "language_detected": "<language name>"
}
No prose, no markdown, only the JSON object.
```

**Step 2 — Priority scoring (your own logic, not the LLM).** Combine:
- urgency mapped to a base score (critical=90, high=70, medium=45, low=20)
- + keyword boosts (e.g. +10 for "trapped"/"unconscious"/"child"/"elderly")
- + a small time-decay boost so older unresolved requests rise over time

This hybrid (LLM judgment + deterministic rules) is a nice thing to point out to judges — it shows you understand LLM outputs shouldn't be blindly trusted for a life-safety ranking.

**Step 3 — Resource matching (LLM call #2, optional but impressive).** Once you've shortlisted the top 3 resources by type + distance, you can optionally ask the LLM to pick the best one and produce a one-line justification ("Assigning Medical Team 2 — closest team with medical capability, 1.8 km away, currently free"). This is what gives your demo an "it's actually reasoning" moment.
